# ADR-005 — Cifrado PHI en aplicación y audit log inmutable

## Estado
Aceptado

## Contexto
La Fase 4 introduce manejo directo de PHI en clinical-service.
El cifrado solo a nivel de disco no protege contra accesos indebidos
desde consultas SQL, dumps lógicos o operadores con acceso de lectura.
Además, se requiere trazabilidad legal de accesos PHI bajo
auditabilidad de tipo append-only.

## Decisión
Se adopta cifrado de aplicación para PHI usando AES-256-GCM con
clave por tenant y payload versionado.

También se define tabla `phi_audit_log` con política insert-only,
revocando `UPDATE` y `DELETE` para `app_clinical_user`.

## Alternativas consideradas
- Cifrado solo a nivel de disco — descartada porque no protege frente
  lectura lógica desde SQL o backups descifrados por el motor.
- Cifrado por columna gestionado solo por DB — descartada porque
  acopla demasiado la lógica de seguridad al motor y dificulta
  portabilidad entre ambientes.

## Consecuencias
- Positivas:
  - PHI permanece cifrado en reposo incluso ante lectura de tablas.
  - Trazabilidad de accesos más robusta con auditoría append-only.
  - El dominio mantiene abstracción vía puertos (`EncryptionPort`,
    `AuditLogPort`).
- Negativas / trade-offs:
  - Mayor complejidad operativa para rotación de claves por tenant.
  - Coste computacional adicional por cifrado/descifrado.

## Trigger de revisión
Revisar esta decisión si:
- se requiere rotación automática sin downtime por tenant,
- se incorpora HSM/KMS con capacidades nativas de envelope encryption,
- aparecen requisitos regulatorios nuevos sobre anonimización/pseudonimización.
